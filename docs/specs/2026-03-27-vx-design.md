# vx — vLLM Model Manager

**Date:** 2026-03-27
**Status:** Draft

## Problem

Managing vLLM models on a GPU workstation requires memorizing long script paths, manually guessing serving parameters, and having no way to compare models against real workloads. The existing bash scripts (`list-models.sh`, `download-model.sh`, `tune-model.sh`) are fragile, untestable, and disconnected.

## Solution

A single Python CLI tool (`vx`) that replaces all existing scripts with a guided, interactive experience. Users describe their workload; `vx` tells them which model to use and how to configure it.

## Tech Stack

- **CLI framework:** Typer (brings click + rich)
- **Package manager:** uv
- **Testing:** pytest
- **Output:** Rich tables, panels, progress bars
- **HF integration:** `huggingface_hub` (already in vLLM venv)
- **Benchmarking:** `vllm bench sweep` (vLLM 0.18+)

## Commands

### `vx` (no args) — dashboard

Replaces `/etc/profile.d/vllm-welcome.sh`. Shows GPU, active server, downloaded models, and available commands at a glance.

### `vx models [model]` — list and inspect

Without args: table of all downloaded models with size, probe status, tune status, max context.

With a fuzzy model name: detailed view showing the 2D limits table (context length x kv-dtype → max concurrency slots) and which profiles have been tuned.

### `vx download [model_id]` — search and download

Without args: interactive HuggingFace search. Uses `HfApi.list_models(search=..., pipeline_tag="text-generation")` to find models, displays results in a table with quant type, size, and download count. User picks one.

With a model ID: direct download via `huggingface_hub.snapshot_download()`.

After download, prompts to run `vx tune` (interactive terminals only, guarded by `sys.stdin.isatty()`).

### `vx tune <model> [profile]` — probe limits + benchmark

Two phases:

**Phase 1: Probe (always runs first, ~15 min, cached)**

Binary-search context limits for each kv-cache-dtype (auto, fp8):
- For each context length (4k, 8k, 16k, 32k, 64k, 128k, 256k, capped by model's max_position_embeddings):
  - Attempt to start vLLM with that `max-model-len`
  - If it starts successfully, test concurrency: binary search `max-num-seqs` (1, 2, 4, 8, 16, 32, 64, 128, 256) to find the max that doesn't OOM
  - If OOM on startup, mark as null and stop searching higher context lengths for this kv-dtype

Results stored as `limits.json` (see Data Model). Skipped if `limits.json` already exists (use `--reprobe` to force).

`gpu-memory-utilization` is fixed at `(vram_total_gb - 2) / vram_total_gb` for all probes and benchmarks — on a dedicated inference machine, always max out VRAM with 2 GB overhead.

**Phase 2: Benchmark (optional, ~30-120 min per profile)**

Without a profile arg: interactive picker (Interactive, Batch, Agentic, Creative, All, Skip).

With a profile arg: runs that profile directly.

Four benchmark profiles, all using `vllm bench sweep`:

#### `interactive` — chat, coding, RAG

Optimizes for lowest TTFT with prefix caching. Sweeps context lengths within proven limits.

- Serve params swept: `kv-cache-dtype` [auto, fp8], `enable-prefix-caching` [true], context lengths from probe
- Bench params: `max-concurrency` 1, `random-input-len` [256, 512, 2048], `random-output-len` [128, 256], `num-prompts` 20
- Metric: lowest mean `mean_ttft_ms`

#### `batch` — bulk processing

Optimizes for max throughput. Uses `vllm bench sweep serve_workload` to auto-explore concurrency.

- Serve params swept: `max-num-seqs` [8, 16, 32, 64], `max-num-batched-tokens` [1024, 2048, 4096], `kv-cache-dtype` [auto, fp8]
- Bench params: `random-input-len` 512, `random-output-len` 256, `num-prompts` 50
- Metric: highest `output_throughput`

#### `agentic` — tool calling, multi-step

Optimizes for lowest TTFT at maximum context with prefix caching critical.

- Serve params swept: `kv-cache-dtype` [auto, fp8], `enable-prefix-caching` [true]
- Fixed: context at max proven limit from probe
- Bench params: `max-concurrency` 1, `random-input-len` at 75% of max context, `random-output-len` 512, `num-prompts` 10
- Metric: lowest mean `mean_ttft_ms`

#### `creative` — long-form writing

Optimizes for lowest TPOT (decode speed) during long generation.

- Serve params swept: `kv-cache-dtype` [auto, fp8], `max-num-seqs` [1, 2, 4, 8]
- Fixed: context at max proven limit from probe
- Bench params: `max-concurrency` 1, `random-input-len` 256, `random-output-len` [1024, 2048, 4096], `num-prompts` 5
- Metric: lowest mean `median_tpot_ms`

### `vx compare` — find the best model for a workload

Interactive guided flow:
1. How many concurrent users?
2. Use case? (Interactive / Batch / Agentic / Creative)
3. Context needed? (tokens)

Reads `limits.json` for each downloaded model (runs probe first if missing). Filters models that can handle the requested concurrency at the requested context. Ranks by benchmark metric if available, otherwise by capacity.

Outputs a recommendation with the `vx start` command to use.

Also works non-interactively: `vx compare --users 10 --use-case interactive --context 8000`

### `vx start [model] [--profile]` — start serving

Without args: interactive model picker → profile picker → launch.

Starts vLLM via systemd or direct `vllm serve` depending on whether a tuned config exists:
- Tuned config exists → `vllm serve --config <profile>.yaml`
- No tuned config but probe exists → generate safe defaults from limits
- Nothing exists → prompt to run `vx tune` first

Updates the `active.yaml` symlink.

Non-interactive: `vx start nvfp4 --profile interactive`

### `vx stop` — stop serving

Stops the vLLM systemd service. Shows what was running.

## Fuzzy Model Matching

All commands that accept a model name use fuzzy matching against downloaded models:

1. Exact match on full path: `/opt/vllm/models/apolo13x/Qwen3.5-27B-NVFP4`
2. Exact match on `provider/model`: `apolo13x/Qwen3.5-27B-NVFP4`
3. Case-insensitive substring match on model name: `nvfp4`, `27b-nvfp4`, `huihui`
4. If exactly one match → use it
5. If multiple matches → show them and ask user to pick (interactive) or error with list (non-interactive)
6. If no match → error with suggestion (did you mean?)

## Data Model

All data lives under `/opt/vllm/`:

```
/opt/vllm/
├── models/<provider>/<model>/              # downloaded models
│   └── config.json                         # HF model config (quant, arch, etc.)
├── configs/
│   ├── active.yaml                         # symlink → current serving config
│   ├── .env                                # HF token
│   └── models/
│       ├── <provider>--<model>.limits.json  # probe results
│       ├── <provider>--<model>.interactive.yaml
│       ├── <provider>--<model>.batch.yaml
│       ├── <provider>--<model>.agentic.yaml
│       └── <provider>--<model>.creative.yaml
├── benchmarks/<model>/<experiment>/        # raw vllm bench sweep output
└── venv/                                   # vLLM installation
```

## Model Property Detection

Read from `config.json` in the model directory:

| Property | Source | Use |
|---|---|---|
| Quantization | `quantization_config.quant_method` | Fixed `--quantization` flag |
| Architecture | `architectures[0]` | Detect MoE |
| MoE | `text_config.num_experts` or `num_experts` | Cap max-num-seqs in batch profile |
| Max positions | `text_config.max_position_embeddings` or `max_position_embeddings` | Cap context probe |
| Model size | Sum of `*.safetensors` file sizes | Informational |

Quantization method → vLLM flag mapping:

| `quant_method` | vLLM flag |
|---|---|
| `gptq` | `--quantization gptq_marlin` |
| `awq` | `--quantization awq_marlin` |
| `fp8` | `--quantization fp8` |
| `compressed-tensors` | (none — auto-detected) |
| absent / null | (none) |

## OOM Handling

### During probe

OOM = binary search termination point. Expected and handled: try to start, if exit code != 0 within timeout, mark as null, move on.

### During benchmark

`vllm bench sweep` exits non-zero on OOM. The tune command catches this and retries with `--resume` up to 3 times. Partial results are usable.

## Testing Strategy

- **Unit tests** use `tmp_path` fixtures with fake `config.json` files, fake `limits.json`, and mock subprocess calls. No real GPU needed.
- **Fuzzy matching** tested with parametrized inputs.
- **Profile generation** tested by asserting JSON structure and value bounds.
- **Winner selection** tested with synthetic benchmark data.
- **Integration tests** marked `@pytest.mark.slow`, skipped by default, actually start vLLM.

## Installation

```bash
cd ~/vx
uv sync
uv pip install -e .
sudo ln -sf $(which vx) /usr/local/bin/vx
```

Update `/etc/profile.d/vllm-welcome.sh` to call `vx` instead of the gum-based script.
